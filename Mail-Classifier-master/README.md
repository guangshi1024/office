# 基于内容的邮件多分类器

如今，人们每天在工作或生活中都会被大量涌入的邮件所淹没。海量的邮件不断引发困扰。邮件分类可以应用于多种不同的场景，包括根据优先级过滤邮件、将邮件分配到用户创建的文件夹，或识别垃圾邮件。不仅是垃圾邮件被视为"无用邮件"，一些不受欢迎的邮件（如广告）也会浪费人们的阅读时间。因此，开发可靠的邮件自动分类系统以解决这一困扰变得刻不容缓。

## 目标
根据邮件内容对邮件进行监督式分类，将其归入用户创建的文件夹中，从而基于已分类的邮件构建一个分类体系（并为此创建一个 Gmail 网页应用）。

## 简介
邮件分类属于机器学习中的文本分类领域。多年来，垃圾邮件过滤是其成功应用的典范，已达到了很高的准确率。邮件归档或邮件的自动分类也可以通过使用各种技术来实现。

### 邮件表示
邮件可以表示为特征上的分布，其中特征可以是单词、单词序列、词性标签、词簇等。因此，所有可能的特征均采用词袋模型。文档表示通常经过降维变换处理。

### 分类器选择
多种分类器可应用于文本分类任务。我们选择最适合我们数据并能产生最佳结果的分类器。

## 设计

### 数据集
为了研究和观察邮件归档的各种方法和分类器，我们使用了来自安然公司员工的大量真实邮件语料库。安然语料库是在对安然公司进行法律调查期间公开的。我们使用了由卡内基梅隆大学提供的经过清理和精炼的语料库。该语料库包含七位前安然公司员工的邮件目录，每人拥有超过一千封邮件和相当数量的文件夹。诸如"收件箱"、"已发送"、"垃圾箱"等自动生成的文件夹已被移除。
该数据集共包含约 8000 封邮件。

### 工作流程
我们首先将语料库中的邮件归入五个文件夹用于通用评估。我们将所有邮件分为以下五个类别：
* 个人
* 商务
* 同事
* 广告
* 组织机构

从邮件的"收件人"、"发件人"、"主题"和"正文"中提取数据后，我们进行了词干提取。
现在可以使用单词列表采用多种策略进行特征选择。特征选择方式可以包括使用完整的单词计数、短语计数或词性标签计数。
我们首先使用一个开源的 Python 代码进行词干提取。这将大幅减少特征提取后获得的单词总数，并将不同形式的单词转换为相同的形式。然后，单词计数被存储在两个不同的字典中，分别对应正文和非正文文本。
此步骤的结果给出了数据集中出现的所有词元的排序列表及其计数（即在所有邮件文档中出现的次数）。总体而言，共有约 7500 封邮件和近 30000 个词元。

### 修改词元列表
一种方法是将词性标签作为特征并分别统计它们的计数，但我们选择移除那些明显不太重要的词性标签，如冠词、介词、代词等。我们使用 Python 中可用的自然语言工具包来移除这些不必要的单词。这使得词元数量从约 30000 个大幅减少到约 20000 个。这也是因为我们还选择移除了外语单词和一个或两个字符的片段，因为它们虽大幅增加了特征列表，却没有提供相应的收益。即使经过了这一步，仍有大量计数很少的词元，我们可以将它们丢弃，因为它们在其他邮件文档中很少出现。另一方面，也有一些词元的计数过高，这些词元在分类任务中信息量不大。最终，词元列表缩减到 1209 个词元，覆盖总共 7562 封邮件。我们还添加了 4 个特殊词元：NUMBER、DOLLAR、HTTP 和 EMAIL。
因此我们总共有 1213 个属性。

### 创建特征向量矩阵
此步骤的主要任务是将每个样本转化为一个特征向量，然后将它们组合成一个完整的矩阵。
* 解析每封邮件，首先识别其标签。
* 然后，存储词元列表中相应元素的计数。
* 特殊单词、表达式和字符被替换为特殊词元 NUMBER、DOLLAR、HTTP 和 EMAIL。
* 将每封邮件的特征向量组合成矩阵后，我们以 80% 的比例将样本划分为训练集和测试集。

### 分类
我们使用朴素贝叶斯、支持向量机（SVM）和随机森林技术对邮件进行分类。SVM 采用一对多方法进行多分类。

## 结果
下表以准确率的形式报告了在七个安然数据集上的结果。

| 用户 | 邮件数量 | 文件夹数量 | 朴素贝叶斯 | SVM | 随机森林 |
|-----|-----|-----|-----|-----|-----|
| beck-s | 1971 | 101 | 28.26 | 41.4 | 48.45 |
| farmer-d | 3672 | 25 | 47.79 | 61.98 | 69.66 |
| kaminski-v | 4477 | 41 | 38.44 | 47.78 | 56.42 |
| kitchen-l | 4015 | 47 | 35.12 | 48.97 | 57.36 |
| lokay-m | 2489 | 11 | 61.55 | 65.45 | 76.66 |
| sanders-r | 1188 | 30 | 53.37 | 61.36 | 65.66 |
| williams-w3 | 2769 | 18 | 72.44 | 84.33 | 89.82 |

在拥有一到两个主导文件夹的数据集上，准确率较高。lokay-m 和 williams-w3 正是这种情况。实际上，williams-w3 仅由两个文件夹组成，这极大地便利了分类过程，从而提高了准确率。显然，只有一个最可能的类别时，分类是一项简单的任务，这也解释了准确率为何如此之高。人们可能期望分类性能会随着训练集大小的增加而提升，但在实践中这很少发生。这可以通过一个观察来解释：邮件通常与其他最近收到的邮件相关，而非与很久以前收到的邮件相关。因此，训练集中较早的邮件可能不会对分类过程产生影响。

我们还生成了一个合并数据集，按照前文设计中的五个常用文件夹进行分类。平衡数据集和仅分组邮件的分类结果分别展示如下。

| 数据类型 | 邮件数量 | 文件夹数量 | 朴素贝叶斯 | SVM | 随机森林 |
|----------|----------|----------|----------|----------|----------|
| 不平衡 | 9853 | 5 | 56.87 | 63.82 | 75.67 |
| 平衡 | 7562 | 5 | 60.58 | 67.42 | 79.78 |

分类结果出人意料地偏低，这说明了邮件分类任务的复杂程度，与常规文本分类截然不同。

## 实现
我们决定将想法转化为一个可运行的模型，因此我们筛选了可以使用的可能工具。

可选项：
* Google App Script
* Gmail Gadgets
* Chrome 扩展

### 我们的选择

在深入了解了各种方案及其特性后，我们决定使用 GAS（Google App Script）。我们否定了制作 Chrome 扩展的想法，因为它不支持 Gmail API，因此我们无法使用 Chrome 扩展为收到的邮件分配标签。同时，我们也否定了使用 Gmail Gadgets 的想法，因为我们发现 GAS 拥有更多实现我们想法所需的工具。

在开始用 GAS 实现时，我们在设计上遇到了以下不足：
* 在此之前我们的工作是使用 Python 完成的，但现在我们不得不在 Javascript 中寻找库和资源，因为 GAS 不支持 Python。
* 我们尝试使用 pyjs 将 Python 与 Javascript 集成，但发现在 GAS 下使用包非常困难。
* 然后我们尝试在本地服务器上运行 Python，但 Google 不允许我们从 App Script 向本地服务器发送数据。
* 最后，我们尝试做出一些取舍，仅用 Javascript 构建我们的模型。

我们发现，Javascript 中可用于实现机器学习的库仅支持二分类。

因此，我们决定制作一个我们构想的原型，将邮件分为两个类别，即个人邮件和工作邮件。

我们决定实现 SVM，经过搜索，我们在 GitHub 上找到了一段可以在我们实现中使用的 Javascript 代码。我们编写了代码，用于访问用户的 Gmail 账户、扫描已分类的邮件、使用 SVM 训练模型并对未分类的邮件进行分类。
然后，我们决定将其实现为一个独立的网页应用，用户可以通过在地址栏中输入其 URL 来访问。

## 未来改进方向

如上所述，我们需要大幅提升准确率。为此，我们必须拥有优质的数据集，并可以改进我们用于分类数据的各种技术。
* 我们可以将此实现推广到任何用户。
* 我们可以实现将邮件分类为多个类别，而不仅仅是二分类。
* 另外结合使用各种采样和回归方法以提高准确率。
* 更充分地利用自然语言处理技术，以便更轻松地区分不同类别。
